Memory Structure Implementation Notes

Design decisions

  • Removed the external memory proposal HTTP routes before replacing the backend so proposal review is no longer part of the supported runtime surface.
  • Removed the bridge-layer workspace.memory.* native job handlers as an explicit cut toward a tree-native backend with fewer legacy entry points.
  • Removed the internal memoryService.capture(...) API because it only existed to support the old proactive context capture flow, which is no longer part of the supported memory direction.
  • Removed the proactive workspace context HTTP route and native bridge job path rather than preserving them on top of the new memory work.
  • Rewrote the PRD around one memory root with two ownership categories, interaction and integration, replacing the earlier topic and daily_digests framing.
  • Locked the node model so every persisted node body is markdown, source leaves are bottom-level documents with no children, and summary nodes are markdown documents that summarize either source leaves or lower summary nodes.
  • Added an explicit ingestion vocabulary to the PRD: event is the raw incoming memory signal, chunk is the normalized admission-stage unit, and source leaf is the durable markdown node created only after a chunk is admitted.
  • Locked the topology vocabulary to memory root -> tree category -> tree, and within each tree to root summary node -> intermediate summary nodes -> source leaves.
  • Chose a no-shared-leaves design. Each leaf belongs to exactly one tree, and cross-category recall is handled by the retrieval layer rather than by multi-tree leaf membership.
  • Attached date and time metadata to every node while explicitly keeping date out of the ownership hierarchy for v1.
  • Locked the lifecycle direction to event -> chunk -> source leaf -> summary node, with chunks treated as transient admission-stage units and leaves treated as durable bottom-level nodes.
  • Locked the write-lifecycle preference so admitted source leaves are treated as immutable, and new information should usually create a new leaf with supersession metadata rather than rewriting old leaf bodies in place.
  • Chose an LLM-driven interaction entity assignment path for v1, where the model decides whether a chunk fits an existing entity or needs a new one, while backend code still owns canonical id generation, validation, and fallback routing.
  • Added an explicit interaction entity decision matrix: low admission confidence rejects the chunk, high existing-match confidence attaches to an existing entity, high new-entity confidence mints a new entity, and otherwise the durable leaf falls back to interaction:uncategorized.
  • Simplified phase 1 to interaction-only implementation. Integration trees remain part of the target architecture but are deferred from the first build.
  • Removed hotness from phase 1. An interaction entity becomes a tree as soon as its first admitted leaf is assigned, and summary layers are created only by seal thresholds.
  • Added a concrete phase-1 upsert algorithm covering chunk admission, LLM entity assignment, backend validation, duplicate no-op handling, superseding leaf creation, tree creation, and summary invalidation/rebuild.
  • Added a retrieval design where durable memory access becomes an agent tool backed by the tree query engine, with summary-first traversal used for broad recall and leaf drill-down used for exact evidence.
  • Explicitly chose to embed sealed intermediate summary nodes as well as leaves, using summary embeddings for branch selection and leaf embeddings for final evidence retrieval.
  • Updated phase-1 summary generation so interaction summary nodes use an LLM-authored summary sentence when a summary model client is available, while keeping deterministic node ids, deterministic wrapper markdown, and deterministic fallback behavior when no model client is available.
  • Implemented phase 1 by introducing dedicated interaction-memory tables in the runtime state store for entities, leaves, summary nodes, tree edges, and node embeddings, instead of reusing the legacy memory_entries catalog.
  • Cut runtime durable writeback and bootstrap recall directly over to the new interaction-memory module, so completed turns now persist interaction leaves and runner bootstrap now recalls interaction hits without going through MEMORY.md planning.
  • Added a new runtime agent tool, memory_retrieve, as the supported agent-facing entry point for durable memory retrieval in phase 1.
  • Removed the desktop workspace-ready heartbeat path entirely, including the proactive ingest emit and its earlier runtime-captured context dependency, so workspace creation no longer depends on legacy proactive bootstrap behavior.

Deviations

  • Left the underlying proposal record types and proposal creation internals in place for now. This change removes the exposed APIs first and avoids coupling the initial cleanup to the larger backend rewrite.
  • Kept /api/v1/memory/sync in place for now even though the current filesystem backend only returns a success wrapper around status. That route can be removed or redefined once the new backend contract is in place.
  • Kept the generic file-oriented FilesystemMemoryService surface for search, get, and upsert so other workspace features that still write markdown artifacts can continue to use it while durable interaction memory lives in the new tree backend.
  • Used deterministic summary-node bodies derived from child titles and summaries for phase 1 instead of invoking an LLM to author tree summaries. This keeps sealing synchronous and testable while the tree structure settles.
  • Replaced the old recall-selection model flow entirely. Runner bootstrap now retrieves directly from interaction leaves and summaries, and the only model call in recall is the optional embedding request.
  • Interaction-memory retrieval currently uses the default recall embedding model (text-embedding-3-small) rather than the runtime background-tasks chat model. That keeps retrieval on a stable embedding model even when the execution model differs.
  • Summary-node bodies are still structurally wrapped by backend templates. The LLM currently supplies the summary prose for each summary node, not the full markdown file.
  • Workspace creation no longer emits the legacy proactive heartbeat at all. This intentionally removes a bootstrap side effect that was only used by the deprecated proactive proposal flow.

Tradeoffs

  • This keeps the first step small and testable, but it means there is still dormant proposal-related backend code that will need removal in a later pass.
  • The PRD now prefers strict tree ownership over multi-tree projections. That simplifies storage and tree semantics, but it pushes more cross-category composition into the retrieval layer.
  • Keeping the generic file-memory abstraction alongside the new interaction backend makes the transition much less disruptive for unrelated artifact-writing flows, but it leaves the public /api/v1/memory/* routes semantically broader than the tree-native durable backend they now report on.
  • Deterministic summaries avoid another model dependency in the hot path, but they produce flatter summary prose than a dedicated summarization model would.
  • Using an LLM only for the summary prose raises summary quality without making node identity unstable, but it still adds model latency and possible variance whenever tree rebuilds run with a summary model client.

Open questions

  • The PRD intentionally defers the exact granularity of integration trees, future hotness rules for interaction-tree materialization, the long-term fate of the generic /api/v1/memory/* routes, and whether later phases should upgrade summary generation from deterministic templates to model-written summaries.
