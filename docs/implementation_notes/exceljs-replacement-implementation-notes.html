ExcelJS Replacement Implementation Notes

Design decisions

  • Replaced exceljs with xlsx-populate for .xlsx load/edit/save paths because it avoids the legacy exceljs -> unzipper -> fstream packaging chain and is designed to preserve existing workbook structure when re-saving.
  • Kept CSV handling out of the spreadsheet library replacement and moved it to local parser/serializer helpers so desktop spreadsheet previews no longer depend on workbook-level CSV support.
  • Used createRequire(__filename) in the Electron main process so the packaged CommonJS main bundle can load xlsx-populate without relying on ESM-only import.meta.url behavior.

Deviations

  • The replacement preserves the existing .xls behavior instead of expanding support. Legacy .xls files remain preview-only in the inline editor.
  • Cell display text for some advanced Excel formats now comes from normalized raw workbook values instead of exceljs's formatted cell.text output.

Tradeoffs

  • xlsx-populate is a better fit for in-place workbook edits and packaging compatibility, but it exposes less built-in display formatting than exceljs.
  • Local CSV parsing is narrower than a full CSV package, but it removes another transitive dependency path and matches the current inline editor scope.
  • The runtime bundle prune step now removes dangling workspace symlinks after stripping .d.ts and sourcemap files, which keeps Electron packaging stable but makes the pruning logic slightly more opinionated about broken links in staged output.

Open questions

  • If richer Excel display formatting becomes important in previews, we may want a dedicated formatting layer on top of the workbook reader instead of depending on workbook-library display strings.
